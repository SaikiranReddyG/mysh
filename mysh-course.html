What is mysh? The Shell as an Orchestra Conductor

Understanding what shells do and why building one teaches you about the Unix philosophy

Meet the Shell: Your Computer's Concierge

When you open a terminal and type ls, you're not directly talking to your computer. You're talking to a shell.

The shell is like a concierge at a fancy hotel. You ask for something in English ("show me the files"). The concierge doesn't do the work themselves—they find the right department ("the file listing department"), tell them what you want, wait for the result, and bring it back to you.

mysh is a shell. It's a working implementation of this concierge, built from scratch in C using only raw system calls.

What Shells Actually Do

📝
Read

Wait for you to type a command

⚙️
Parse

Break the command into pieces (program name, arguments, pipes, redirects)

🚀
Execute

Start the program and let it run

📊
Output

Show you the results, then wait for the next command

Why You Should Care About Shells

💡
It's Not Just About Running Commands

Shells teach you how your computer actually works. Understanding processes, file descriptors, signals, and job control isn't shell-specific — it's the foundation of all systems programming. When you understand mysh, you understand the operating system.

A Real Example: What Happens When You Type

Imagine you type: ls | grep .c

Here's what happens under the hood:

1
Parse the input

Shell reads the command and sees two programs: ls and grep, connected by a pipe (|)

2
Create a pipe

Shell makes a communication channel — a pipe — that will connect ls's output to grep's input

3
Fork twice

Shell creates two child processes: one for ls, one for grep

4
Connect the pipes

ls's stdout is connected to the pipe, grep's stdin is connected to the pipe

5
Execute

Both programs run. ls outputs file names, grep filters them, and you see only .c files

Meet the Cast of Characters

Each piece of mysh has a specific job. Understanding who does what helps you steer AI and debug problems.

The Main Story: REPL (Read-Eval-Print Loop)

The heart of mysh is a loop that repeats four times per command:

CODE 
while (!should_exit) {
  printf("mysh> ");

  pipeline = parse_input(buffer);
  history_add(buffer);

  if (is_builtin(...)) {
    execute_builtin(pipeline);
  } else {
    pid = execute_command(...);
    waitpid(pid, NULL, 0);
  }
}
WHAT IT DOES

Read: Show the prompt and get what the user typed

Eval: Parse it and check if it's a built-in command

Execute: Either run the built-in or fork a new process for external programs

Print: Wait for it to finish, then loop back for the next command

The Seven Players (Modules)

📖

Parser

Breaks `ls -la | grep x > file` into tokens, detects pipes, finds redirects

🚀

Executor

Forks a new process and runs a program using execvp()

🔗

Pipes

Wires up multiple commands so output of one feeds input to next

📂

Redirects

Sends stdout to files (>), reads stdin from files (<), captures errors (2>)

⌨️

Built-ins

Special commands (cd, exit, jobs) that run inside the shell, not as separate programs

🎯

Signals

Handles Ctrl+C and Ctrl+Z so they interrupt programs, not the shell itself

📋

Jobs

Tracks background processes so you can bring them back with fg

When you type `sleep 10 &` and press Enter:

Why This Architecture Matters

ℹ️
Separation of Concerns

Each module does one job and does it well. The parser doesn't execute, the executor doesn't parse. This makes the code easier to understand, modify, and debug. When something breaks, you know exactly where to look.

Parsing: Breaking Commands into Pieces

How mysh reads your input and understands what you mean

The Parsing Problem

You type: `ls -la | grep .c > files.txt`

Your shell needs to understand:

  • Two programs: ls and grep
  • ls gets arguments: -la
  • grep gets arguments: .c
  • They're connected by a pipe: |
  • The output goes to a file: files.txt

If the parser gets this wrong, nothing works. So let's see how it works.

Step 1: Tokenization — Split by Spaces

CODE 
char *saveptr;
char *token = strtok_r(cmd,
                      " \t",
                      &saveptr);

while (token) {
  // save this word
  token = strtok_r(NULL,
                   " \t",
                   &saveptr);
}
WHAT IT DOES

strtok_r is a function that splits a string. It breaks up the command every time it sees a space or tab.

Each piece becomes a "token" — a word the shell understands.

From `ls -la | grep .c`, we get: ["ls", "-la", "|", "grep", ".c"]

Step 2: Detect Special Symbols

After tokenizing, the parser looks for special symbols:

|

Pipe

Two commands talk to each other

>

Redirect Out

Send output to a file

<

Redirect In

Read input from a file

&

Background

Run in background, don't wait

Step 3: Build the Data Structures

Once parsing is done, the shell builds two key structures:

CODE 
typedef struct {
  char *argv[MAX_ARGS];
  int argc;
  Redirect *redirects;
  int bg_flag;
} Command;

typedef struct {
  Command *commands;
  int num_commands;
} Pipeline;
WHAT IT MEANS

Command: One program to run, with arguments, redirects, and whether it's background

Pipeline: One or more commands connected by pipes

For `ls | grep | wc`, the Pipeline has 3 Commands

Execution: The Fork/Exec Magic

How mysh actually runs your programs — the most important syscalls in Unix

The Fundamental Problem

The shell is running. When you type `ls`, the shell needs to run `ls` but also stay alive to accept your next command. How?

The answer: fork and exec.

Fork: Cloning the Shell

CODE 
pid_t pid = fork();

if (pid == -1) {
  // fork failed
}
else if (pid == 0) {
  // I'm the child
}
else {
  // I'm the parent, pid has the child's ID
}
WHAT HAPPENS

fork() creates an exact copy of the shell process.

Now there are TWO processes running the same code.

The parent gets the child's PID and knows about the child. The child gets 0.

They can now go different directions — the parent stays a shell, the child becomes `ls`.

Exec: Becoming ls

CODE 
if (pid == 0) {
  // Child process

  // Apply any redirects first
  apply_redirects(cmd);

  // Replace this process with ls
  execvp(
    "ls",
    argv
  );

  // If execvp returns, it failed
  perror("ls");
  exit(127);
}
WHAT HAPPENS

execvp completely replaces the child process with a new program.

The child's memory, code, everything is overwritten with `ls`.

It's like a costume change — the person doesn't change, but now they're dressed as `ls`.

If `execvp` succeeds, it never returns. The process is now `ls`, not the shell.

Wait: The Parent Holds Its Breath

CODE 
else {
  // Parent process

  // If foreground, wait for child
  if (!is_background) {
    waitpid(pid, NULL, 0);
  } else {
    // Background: add to job list
    add_job(pid, JOB_RUNNING, cmd);
  }
}
WHAT HAPPENS

waitpid tells the parent: "Sit here and wait for child PID to finish."

The shell blocks (does nothing) until `ls` completes.

When `ls` exits, the shell wakes up and shows its prompt again.

If the & flag was used, the shell doesn't wait — it adds the job to a list and keeps going.

Why This Matters for You

💡
The Fork/Exec Pattern Appears Everywhere

This isn't shell-specific. Whenever a program needs to run another program (a web server spawning a worker, a scheduler launching a task, a container engine starting containers), fork/exec is the pattern. Understanding it here means you understand a fundamental building block of computing.

Pipes: How Programs Talk to Each Other

The secret sauce that makes Unix powerful — anonymous communication channels

What is a Pipe?

A pipe is not a file. It's not on disk. It's an anonymous channel that connects two processes in memory.

Think of it like a post office mail slot: one process drops data in the slot (writing to the pipe), another picks it up (reading from the pipe). Once you read data, it's gone — the next read gets different data.

Try this: `ls | wc -l`

ls outputs file names. wc -l counts lines. The pipe is what connects them—ls's output becomes wc's input.

Creating a Pipe: Two File Descriptors

File descriptors are just numbers (0, 1, 2...) that point to open resources. When you create a pipe, you get two file descriptors:

CODE 
int pipefd[2];
pipe(pipefd);

// pipefd[0] = read end
// pipefd[1] = write end
WHAT IT MEANS

pipe() creates an anonymous channel in memory.

The first FD (pipefd[0]) is for reading from the pipe.

The second FD (pipefd[1]) is for writing to the pipe.

They're connected — whatever you write to [1] you can read from [0].

Wiring Up: dup2() Magic

After forking, each child process has a copy of the file descriptors. The shell uses dup2() to "rewire" them:

CODE 
for (int i = 0; i < num_commands; i++) {
  pid = fork();

  if (pid == 0) {
    // Child: connect my stdin to pipe
    if (i > 0)
      dup2(pipes[i-1][0], 0);

    // Connect my stdout to pipe
    if (i < num_commands-1)
      dup2(pipes[i][1], 1);

    execvp(...);
  }
}
WHAT HAPPENS

dup2(fd, newfd) redirects file descriptor newfd to point to fd.

dup2(pipes[i-1][0], 0): "Make my stdin (fd 0) point to the pipe I need to read from"

dup2(pipes[i][1], 1): "Make my stdout (fd 1) point to the pipe I need to write to"

Now when the child writes to stdout, it goes into the pipe. Pretty clever!

Signals and Background Jobs

How mysh handles Ctrl+C, Ctrl+Z, and lets you run programs in the background

The Problem: Interrupting Programs

You're running `sleep 100`. You get bored. You press Ctrl+C.

What should happen?

  • The sleep process should stop (not the shell)
  • The shell should display its prompt again
  • You should be able to type a new command

If the shell didn't handle signals carefully, Ctrl+C would kill the entire shell. You'd lose your work.

Signals: Interrupts from the OS

A signal is how the OS tells a process "something happened." Ctrl+C sends SIGINT. Ctrl+Z sends SIGTSTP.

⚡

SIGINT (Ctrl+C)

Interrupt — stop what you're doing

⏸️

SIGTSTP (Ctrl+Z)

Stop — pause and let me do something else

☠️

SIGCHLD

Child died — clean up the dead process

Shell Strategy: Ignore Signals

The shell does something clever: it ignores SIGINT and SIGTSTP.

CODE 
void setup_signals() {
  struct sigaction sa;
  sa.sa_handler = SIG_IGN;

  sigaction(SIGINT, &sa, NULL);
  sigaction(SIGTSTP, &sa, NULL);
}
WHAT IT DOES

When the shell starts, it registers signal handlers.

SIG_IGN means "ignore this signal — don't do anything."

Now when you press Ctrl+C, the shell doesn't die. But the child process does (children inherit the default handlers).

Background Jobs: Run Without Waiting

When you type `sleep 100 &`, the & tells the shell "don't wait for this."

CODE 
if (is_background) {
  add_job(pid, JOB_RUNNING, cmd);
  printf("[%d] %d\n",
         job_id, pid);
} else {
  waitpid(pid, NULL, 0);
}
WHAT HAPPENS

If the & flag was parsed, the shell adds the job to its job list instead of waiting.

It prints the job ID and PID so you know which job you're managing.

Meanwhile, the shell loops back to the prompt ready for another command.

SIGCHLD: Reaping Dead Children

Background jobs don't wait. But the shell eventually learns they finished when the OS sends SIGCHLD ("child died").

ℹ️
Why "Reaping" Matters

If a child process finishes but the parent doesn't acknowledge it, the child becomes a "zombie" — it's dead but still taking up a process table slot. The shell must handle SIGCHLD and call waitpid() to reap the child.

I/O Redirection: Sending Data to Files

How > and < work, and why it's beautiful Unix design

Input/Output: The Three Standard Streams

Every Unix process has three built-in communication channels:

0
stdin

Where data comes FROM (your keyboard by default)

1
stdout

Where data goes TO (your screen by default)

2
stderr

Where errors go (your screen by default)

But they don't have to point to your screen! That's where redirection comes in.

The Redirection Trick: Open → dup2()

When you type `echo hello > file.txt`, here's what happens:

CODE 
int fd = open(
  "file.txt",
  O_WRONLY | O_CREAT | O_TRUNC,
  0644
);

dup2(fd, 1);

execvp("echo", argv);
WHAT HAPPENS

open() creates or opens the file, returning a file descriptor.

dup2(fd, 1) "redirects" stdout (fd 1) to point to that file.

When echo runs and tries to write to stdout, it's actually writing to the file!

From echo's perspective, nothing changed. It just writes to fd 1. But fd 1 now points to a file.

Variants: Truncate vs Append

>

Redirect Output

echo hello > file.txt — truncate (overwrite) the file

>>

Append

echo hello >> file.txt — add to the end

<

Redirect Input

wc -l < file.txt — read from file instead of keyboard

2>

Redirect Errors

cmd 2> errors.txt — save errors to a file

Why This is Beautiful

💡
The Unix Philosophy: Everything is a File

Files, pipes, sockets, devices — they're all treated the same way. A program doesn't need to know (or care) if it's writing to stdout, a file, or across a network. It just writes to fd 1. The shell handles all the wiring. This is why Unix programs compose so beautifully.

Debugging Patterns: How to Solve Problems

Now that you understand mysh, here's how to debug issues:

🔍

"Pipes hang"

Check redirect.c — you probably didn't close pipe ends in the parent

👻

"Zombie processes"

Check signals.c — SIGCHLD handler not reaping children properly

😵

"Shell dies on Ctrl+C"

Check setup_signals() — shell should ignore SIGINT

🚫

"Command not found"

Check parser.c and executor.c — tokenization or argv might be wrong
